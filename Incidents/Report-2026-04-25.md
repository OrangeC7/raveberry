# Incident report 2026-04-25


Online event web UI became slow, returned Bad Gateway, lost live state updates, and sometimes failed song requests from YouTube or ytdlp. The incident was eventually traced to a combination of several factors.

---

## Phase 1 — Fixes made before the site became stable again

### 1. Hardened YouTube URL parsing

**Problem encountered:**
Invalid or malformed YouTube links could raise:

```text
KeyError: 'missing youtube video id'
```

from `YoutubeSongProvider.get_id_from_external_url()`.

**Fix made:**
Changed YouTube ID extraction so bad YouTube URLs are rejected as normal user facing request errors instead of becoming unhandled server exceptions.

**Result:**
Malformed YouTube links will fail gracefully.

---

### 2. Hardened yt-dlp failure handling

**Problem encountered:**
yt-dlp could fail on age restricted, unavailable, or otherwise blocked videos. Logs showed errors such as:

```text
Sign in to confirm your age
This video is not available
```

**Fix made:**
Added handling for `yt_dlp.utils.ExtractorError` and `yt_dlp.utils.DownloadError` during availability checks and downloads. Age restricted videos now set a controlled error like:

```text
Sorry, this video is age restricted.
```

**Result:**
Ravefurry will reject the song gracefully.

---

### 3. Added bounded external request behavior for YouTube lookups

**Problem encountered:**
YouTube/YTMusic requests could hang or take too long, contributing to request pileups.

**Fix made:**
Added explicit timeouts/retry limits around YTMusic/yt-dlp calls.

**Result:**
External YouTube failures won't stall server work indefinitely.

---

### 4. Prevented background request tracking leaks

**Problem encountered:**
The request tracker incremented `active_requests` but could fail to decrement it if a request crashed or was killed.

**Fix made:**
Wrapped tracked request cleanup in `try/finally`.

**Result:**
Runtime request counters will no longer get stuck high after errors.

---

### 5. Reduced `/ajax/musiq/own-song-state/` cost

**Problem encountered:**
`own_song_state` was repeatedly polling per user and could become expensive as users and queue entries increased.

**Fix made:**
Changed ownership checks to use faster requester/queue slot paths instead of repeatedly doing unnecessary per song lookup work.

**Result:**
The “my song” polling endpoint should produce less backend load.

---

### 6. Added frontend state request throttling

**Problem encountered:**
The frontend could fire overlapping `/ajax/musiq/state/` requests when state updates slowed down.

**Fix made:**
Added a `stateRequestInFlight` guard around `getState()`.

**Result:**
The browser should not stack multiple simultaneous state requests from the same page.

---

### 7. Disabled websocket state push as the primary update mechanism

**Problem encountered:**
Logs repeatedly showed `/state/` websocket tasks getting killed:

```text
WebSocketProtocol ... path=b'/state/' took too long to shut down and was killed
```

**Fix made:**
Moved frontend live updates to HTTP polling instead of relying on websocket push.

**Result:**
The UI no longer depends on unstable websocket state delivery.

---

### 8. Restored `/state/` as a harmless compatibility websocket

**Problem encountered:**
After removing the websocket route, cached/old clients still tried to connect and caused:

```text
ValueError: No route found for path 'state/'.
```

**Fix made:**
Restored the `/state/` route with a dummy `StateConsumer` that accepts connections but does not join groups or push state.

**Result:**
Old cached clients no longer create routing exceptions or reconnect storms.

---

### 9. Identified and fixed PostgreSQL connection exhaustion

**Problem encountered:**
The decisive error was:

```text
FATAL: remaining connection slots are reserved for roles with the SUPERUSER attribute
```

This explained why unrelated routes, page loads, state polling, and song queue operations all started failing together.

**Fixes made:**

Changed:

```python
SESSION_SAVE_EVERY_REQUEST = True
```

to:

```python
SESSION_SAVE_EVERY_REQUEST = False
```

Added aggressive DB connection cleanup middleware.

Cached the visitor counter in `base.state_dict()` so state polling no longer repeatedly calls:

```python
Counter.objects.get_or_create(...)
```

**Result:**
The app should no longer burn through PostgreSQL connections under normal polling/event traffic.

---

### 10. Rebooted to clear already exhausted runtime state

**Problem encountered:**
Reinstalling/restarting the app did not immediately fix the system because PostgreSQL and old runtime processes were already wedged.

**Fix made:**
Full system reboot.

**Result:**
PostgreSQL connection slots were cleared, old Python/Daphne/task state was removed, and the patched app began working mostly normally.

---

# Phase 2 — Post recovery bug fixes after the site was working

### 11. Fixed “forgot my song” after idle tabs

**Problem encountered:**
After users left the tab idle or switched away, the frontend sometimes stopped highlighting their own queued song.

**Root cause identified:**
Requester ownership depended too heavily on session/runtime state. If it became inconsistent, the app could forget ownership even though the queued song still existed.

**Fix designed:**
Persist requester identity on the actual song records:

```python
QueuedSong.requester_ip
QueuedSong.requester_session_key
CurrentSong.requester_ip
CurrentSong.requester_session_key
```

**Result:**
Ownership can be recovered from the database instead of depending only on in memory session data.

---

### 12. Fixed “Unknown IP” in moderator queue display

**Problem encountered:**
Some queue entries showed requester IP as unknown in the moderator UI.

**Root cause identified:**
Moderator state used:

```python
user_manager.get_song_requester_ip(queue_key)
```

which previously read in memory data only.

**Fix designed:**
Updated requester lookup so in memory data remains the fast path, but database fields are the fallback.

**Result:**
Moderator UI will still show requester IP even if in memory state is missing.

---

### 13. Preserved requester identity when queued song becomes current song

**Problem encountered:**
When a song moved from `QueuedSong` to `CurrentSong`, the queued row was deleted. If requester identity only existed on the queued row or in memory, current song ownership could be lost.

**Fix designed:**
Copy requester fields into `CurrentSong` during playback transition.

**Result:**
Current song ownership and requester IP will remain available after playback starts.

---

### 14. Added database migration for requester identity fields

**Problem encountered:**
The new persistent requester fields require schema changes.

**Fix designed:**
Added migration:

```text
0019_store_requester_identity.py
```

with requester fields on both `QueuedSong` and `CurrentSong`.

**Result:**
Requester identity becomes durable across idle tabs, in memory misses, current song transitions, and moderator state refreshes.
